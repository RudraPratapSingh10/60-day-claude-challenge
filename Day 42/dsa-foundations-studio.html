Module 01 · Foundational

Arrays — The Foundation

Before you can search or sort, you need a precise mental model of what an array actually is — not just "a list," but a contiguous block of memory you can index into instantly.

1.1 What an array really is

An array is a fixed-size, contiguous block of memory that stores elements of the same type back-to-back. Because every element occupies the same number of bytes, the computer can jump directly to any position using simple arithmetic: address = base_address + index × element_size. That's why reading arr[i] takes the same amount of time whether i is 0 or 999,999 — this is called O(1) random access.

Analogy

Think of an array like a street of identical, numbered houses built in one continuous row. If you know house #7 is exactly 7 plot-widths from house #0, you can walk straight to it without checking every house along the way. A linked list, by contrast, is like a treasure hunt — each house only tells you where the next one is, so you must visit them in order.

Each cell shows its value, its index, and its (simulated) memory address — notice the address increases by a fixed step.

1.2 Core operations & their cost

Common Misconceptions
  • "Arrays can grow freely." Raw arrays are fixed-size; languages simulate growth with dynamic arrays that secretly allocate a new, bigger block and copy everything over.
  • "Insertion is always O(1)." Only appending to the end (with spare capacity) is O(1). Inserting at the front means shifting every other element.
  • "Index 1 is the first element." Most languages (C++, Java, Python, JS) are 0-indexed — the first element is arr[0].
Key Takeaways
  • Arrays trade flexibility for speed: O(1) access, but O(n) insertion/deletion in the middle.
  • Contiguous memory is what makes constant-time indexing possible.
  • Dynamic arrays (vectors/lists) simulate resizing by reallocating and copying.
Module 02 · Applying the Foundation

Searching Techniques

Now that you know how arrays store data, let's find things inside them — starting with the naive approach and building up to a technique that's exponentially faster.

2.1 Linear Search — checking every house

Linear search walks through the array from the start, comparing each element to the target. It works on any array, sorted or not, but in the worst case it checks every single element.

Analogy

You're looking for a friend's house on a street where the numbers aren't in order. You have no choice but to walk past every house and check the number.

Best caseWorst caseRequires sorted?
O(1)O(n)No

2.2 Binary Search — divide and conquer

If the array is sorted, you don't need to check every element. Look at the middle element: if it's your target, you're done. If your target is smaller, it must be in the left half — so throw away the right half entirely. If larger, throw away the left half. Repeat.

Analogy

This is exactly how you find a word in a paper dictionary. You don't start at "A" — you flip to the middle, see you're too far or not far enough, and repeat on the correct half, discarding the rest.

Because each step eliminates half the remaining elements, binary search runs in O(log n) — searching 1,000,000 elements takes at most ~20 comparisons.

Common Misconceptions
  • "Binary search works on any array." The array must be sorted first — sorting itself costs O(n log n).
  • "mid = (low + high) / 2 is always safe." In languages with fixed integer sizes, low+high can overflow; mid = low + (high - low) / 2 is the robust form.
  • "Binary search only finds exact matches." A whole family of problems — "binary search on the answer" — uses it to find boundaries (smallest value satisfying a condition), not just exact targets.
Key Takeaways
  • Linear search: simple, works anywhere, O(n).
  • Binary search: needs sorted data, O(log n), eliminates half the search space each step.
  • The mid-point calculation and loop boundaries are where most bugs live — practice them carefully.
Module 03 · Practical Application

Sorting Algorithms

Binary search needs sorted data — so let's learn how to produce it. We'll go from simple O(n²) approaches to the O(n log n) algorithms used in production.

3.1 The simple family: Bubble, Selection, Insertion

Bubble Sort repeatedly compares adjacent elements and swaps them if out of order, letting the largest values "bubble up" to the end each pass. Selection Sort repeatedly finds the minimum of the unsorted portion and places it at the front. Insertion Sort builds the sorted array one element at a time, inserting each new element into its correct position among those already sorted.

Analogy

Insertion sort is how most people sort a hand of playing cards — you pick up one card at a time and slide it into the correct spot among the cards you're already holding.

3.3 The efficient family: Merge Sort & Quick Sort

Merge Sort splits the array in half recursively until pieces have one element, then merges sorted halves back together — guaranteed O(n log n), stable, but uses O(n) extra space. Quick Sort picks a "pivot," partitions the array so smaller elements go left and larger go right, then recursively sorts each side — average O(n log n) in-place, but worst case O(n²) on already-sorted input with a poor pivot choice.

Analogy

Merge sort is like sorting two already-sorted stacks of exam papers by repeatedly taking whichever top paper has the lower ID — simple to merge once each stack is sorted. Quick sort is like organizing a bookshelf by picking one book, putting everything shorter to its left and taller to its right, then repeating on each side.

Common Misconceptions
  • "Quick sort is always the fastest." It has the best average-case performance, but a poor pivot strategy degrades it to O(n²) — merge sort's guarantee makes it safer for critical systems.
  • "Sorting never needs extra memory." Merge sort needs O(n) auxiliary space; only some algorithms (bubble, selection, insertion, in-place quicksort) are O(1) space.
  • "Stability doesn't matter." It matters whenever you sort by one key but need to preserve the relative order of another (e.g., sorting orders by date after already sorting by customer name).
Key Takeaways
  • Simple sorts (O(n²)) are easy to write and fine for small/nearly-sorted data.
  • Merge sort guarantees O(n log n) at the cost of extra space; quick sort is usually faster in practice but has a bad worst case.
  • Choosing a sort means trading off time, space, and stability for your specific use case.
Module 04 · Mastery

Combining Arrays, Search & Sort

Real interview problems rarely test one technique in isolation. This module shows how sorting and searching combine to solve problems efficiently.

4.1 Pattern: sort first, then search

Many problems become trivial once the array is sorted. Example — Two Sum (sorted variant): given a sorted array, find two numbers that add up to a target. Instead of checking all O(n²) pairs, use two pointers starting at both ends: if the sum is too big, move the right pointer left; if too small, move the left pointer right. This runs in O(n) once sorted.

Why this works
Sorting exposes structure: once ordered, moving a pointer inward is guaranteed to move the sum in a predictable direction — that predictability is what makes two-pointer scanning possible.

4.2 Pattern: binary search on the answer

Sometimes the array you search isn't the input at all — it's the space of possible answers. Example: "What is the minimum capacity needed to ship all packages within D days?" You can binary search over possible capacities (a sorted range of integers) and check, for each candidate, whether it's feasible — shrinking the search space exactly like Module 2's binary search, just applied to answers instead of array indices.

Recognize the pattern when...
The problem asks for a minimum/maximum value satisfying a condition, and "if X works, does X+1 also work?" (monotonicity) holds true.

4.4 Decision guide

SituationTechnique
Unsorted, one-time lookupLinear search — O(n)
Sorted (or can afford to sort once), many lookupsSort once O(n log n), then binary search each O(log n)
Need pairs/triples meeting a sum conditionSort + two pointers
Need the minimum/maximum feasible valueBinary search on the answer space
Common Misconceptions
  • "Sorting always costs too much to be worth it." If you'll query the data many times, paying O(n log n) once is far cheaper than O(n) per lookup.
  • "Binary search only applies to arrays." It applies to any monotonic search space, including answer values, not just physical array indices.
Key Takeaways
  • Sorting is often a pre-processing step that unlocks faster searching.
  • Two-pointer technique turns O(n²) pair problems into O(n) after sorting.
  • Binary search generalizes beyond arrays to any monotonic decision space.

📑 Cheat Sheet

// Binary Search low = 0, high = n-1 while low <= high: mid = low + (high-low)/2 if arr[mid] == target: return mid elif arr[mid] < target: low = mid+1 else: high = mid-1 return -1 // not found

// Bubble Sort for i in 0..n-1: for j in 0..n-i-2: if arr[j] > arr[j+1]: swap(arr[j], arr[j+1])

📝 Summary Notes

Arrays give O(1) access via contiguous memory but O(n) insertion in the middle. Linear search always works; binary search needs sorted data but runs in O(log n). Simple sorts are easy but O(n²); merge and quick sort reach O(n log n) with different space/stability trade-offs. The real skill is recognizing when sorting first, or binary-searching an answer space, turns a slow brute-force approach into an efficient one.
